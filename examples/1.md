This note shows how several agent-memory-shaped concepts can already be modeled in FCP today, and then shows what they would look like if some of them were ever promoted into first-class Fide entity types.

The pattern I see in your examples is:

- treat important real-world artifacts as direct entities
- use statements to connect them
- keep provenance close with statement-to-source links like `prov:wasDerivedFrom`
- use `Action` and `Event` deliberately rather than flattening everything into generic links

That works well for most of the concepts you listed.

## My take first

These concepts break into three groups:

1. Already fits FCP pretty cleanly
- conversation/session
- message/utterance
- tool call
- observation/publication/provenance timing
- event vs action style reasoning artifacts

2. Fits today, but wants stronger conventions
- preference
- claim/assertion confidence
- reasoning trace
- reasoning step

3. Mostly a vocabulary/convention problem, not a protocol problem
- whether you eventually want dedicated terms like `Conversation`, `Message`, `Preference`, `ReasoningTrace`, `ReasoningStep`, `ToolCall`

## Design Rule

Fide entity types are stable base kinds, not contextual roles.

Something should generally become a Fide entity type only when:
- it has stable identity criteria of its own
- it is not just a role played by another entity in a statement
- it would usually be a modeling error, not merely a different modeling choice, to classify it as another existing base type

If something could later still validly be understood as a different base entity type, it is usually better modeled as a statement pattern, qualifier, or domain convention rather than as a new Fide entity type.

That rule is especially important for the examples below.
Several of these are useful application concepts, but that alone does not make them good base entity types for Fide IDs.

Also keep the Fide layers separate:

- the entity type says what kind of thing the entity is
- the reference type says what kind of identifier anchors the Fide ID

So a URL may be used as a `NetworkResource` reference for a `Person`, `Organization`, `Concept`, or other entity type without making the entity itself a `NetworkResource`.

Below I’ll give:
- a plain-English example
- a reasonable FCP modeling choice today
- an FSD-style sketch
- and, where useful, an alternative sketch for if that concept became its own entity type

## 1. Conversation / Session

**Example in my own words**

“A user has an ongoing support conversation with an assistant on April 3. That conversation contains multiple messages and has a start time.”

**Best FCP modeling today**

I would model the conversation/session itself as an `Event`, not a `CreativeWork`.

Why:
- it is a bounded happening in time
- participants can be present in it
- messages can occur within it
- it can have a start/end

If you want the transcript artifact, that transcript could separately be a `CreativeWork`.

**FSD-style sketch today**

```text
# Conversation session as an Event
Event[https://example.com/conversations/support-2026-04-03] schema:name TextLiteral(Customer support conversation on 2026-04-03).
Event[https://example.com/conversations/support-2026-04-03] schema:startDate DateTimeLiteral(2026-04-03T14:00:00Z).
Event[https://example.com/conversations/support-2026-04-03] schema:participant Person[https://example.com/users/alice].
Event[https://example.com/conversations/support-2026-04-03] schema:participant SoftwareAgent[https://example.com/agents/support-bot-v2].
```

**If `Conversation` were a first-class entity type**

I think the difference would mostly be clarity, not expressive power.

```text
Conversation[https://example.com/conversations/support-2026-04-03] schema:name TextLiteral(Customer support conversation on 2026-04-03).
Conversation[https://example.com/conversations/support-2026-04-03] schema:startDate DateTimeLiteral(2026-04-03T14:00:00Z).
Conversation[https://example.com/conversations/support-2026-04-03] schema:participant Person[https://example.com/users/alice].
Conversation[https://example.com/conversations/support-2026-04-03] schema:participant SoftwareAgent[https://example.com/agents/support-bot-v2].
```

This is already pretty clean with `Event`, so I do not think `Conversation` is necessary unless you want a standard domain vocabulary for messaging/agent systems.

## 2. Message / Utterance

**Example in my own words**

“Alice sends the message ‘I need to change my flight’ in that conversation.”

**Best FCP modeling today**

Yes, I think the cleanest split is:

- the message as a whole = `CreativeWork`
- the text content = `TextLiteral`
- the sending = `Action`

That gives you the artifact, the content, and the act separately.

That is actually strong.

**FSD-style sketch today**

```text
# Message artifact
CreativeWork[https://example.com/messages/msg-001] schema:text TextLiteral(I need to change my flight).
CreativeWork[https://example.com/messages/msg-001] schema:author Person[https://example.com/users/alice].
Event[https://example.com/conversations/support-2026-04-03] schema:hasPart CreativeWork[https://example.com/messages/msg-001].

# Sending action
Action[https://example.com/actions/send-msg-001] schema:agent Person[https://example.com/users/alice].
Action[https://example.com/actions/send-msg-001] schema:object CreativeWork[https://example.com/messages/msg-001].
Action[https://example.com/actions/send-msg-001] schema:startDate DateTimeLiteral(2026-04-03T14:01:12Z).
```

**If `Message` were a first-class entity type**

This would make application semantics clearer, but does not unlock much new expressiveness.

```text
Message[https://example.com/messages/msg-001] schema:text TextLiteral(I need to change my flight).
Message[https://example.com/messages/msg-001] schema:author Person[https://example.com/users/alice].
Event[https://example.com/conversations/support-2026-04-03] schema:hasPart Message[https://example.com/messages/msg-001].

Action[https://example.com/actions/send-msg-001] schema:agent Person[https://example.com/users/alice].
Action[https://example.com/actions/send-msg-001] schema:object Message[https://example.com/messages/msg-001].
Action[https://example.com/actions/send-msg-001] schema:startDate DateTimeLiteral(2026-04-03T14:01:12Z).
```

This is one of the cleaner areas in Fide already.

## 3. Preference

**Example in my own words**

“Alice prefers aisle seats.”

**Best FCP modeling today**

This is representable today, but less elegant.

I would not invent a base `Preference` entity unless preferences themselves need stable identity as objects of discussion, revision, provenance, expiration, or conflict handling. If you just want the fact, a statement pattern is enough.

A straightforward current shape is:
- subject = person
- predicate = a preference-like concept or external term
- object = concept or literal

If you want the preference to have provenance, confidence, source, and time, then I would reify the preference as a `Statement` and then attach qualifiers to that statement.

**FSD-style sketch today: simple form**

```text
Person[https://example.com/users/alice] schema:likes Concept[https://example.com/concepts/aisle-seat].
```

**FSD-style sketch today: statement-qualified form**

```text
@1 Person[https://example.com/users/alice] schema:likes Concept[https://example.com/concepts/aisle-seat].
@2 @1 prov:wasDerivedFrom CreativeWork[https://example.com/messages/msg-014].
@3 @1 schema:dateCreated DateTimeLiteral(2026-04-03T14:05:00Z).
```

**If `Preference` were a first-class entity type**

This is where a dedicated type becomes more compelling from an application perspective, but it is also where the Fide design rule pushes back the hardest.

`Preference` often behaves more like:
- a statement pattern
- a qualified claim
- a derived application record

than like a stable base entity kind.

So while this shape is easy to imagine, I would treat it as a convenience-type example, not as an obvious vocabulary recommendation.

```text
Preference[https://example.com/preferences/alice-aisle-seat] schema:name TextLiteral(Alice preference for aisle seats).
Preference[https://example.com/preferences/alice-aisle-seat] schema:about Person[https://example.com/users/alice].
Preference[https://example.com/preferences/alice-aisle-seat] schema:object Concept[https://example.com/concepts/aisle-seat].
Preference[https://example.com/preferences/alice-aisle-seat] prov:wasDerivedFrom CreativeWork[https://example.com/messages/msg-014].
Preference[https://example.com/preferences/alice-aisle-seat] schema:dateCreated DateTimeLiteral(2026-04-03T14:05:00Z).
```

That is semantically heavier, but much better if preference objects need lifecycle and metadata.

## 4. Claim / Assertion Confidence

**Example in my own words**

“A system extracted the claim that Alice likes aisle seats with 0.82 confidence.”

**Best FCP modeling today**

This is where statement reification is doing real work.

The cleanest current move is:
- make the underlying claim a statement
- then talk about that statement as an entity
- attach confidence to the statement, ideally via a separate evaluation action or assertion artifact

You do not want confidence to be confused with the world fact itself.

**FSD-style sketch today**

```text
@1 Person[https://example.com/users/alice] schema:likes Concept[https://example.com/concepts/aisle-seat].
Action[https://example.com/actions/pref-extraction-001] schema:agent SoftwareAgent[https://example.com/agents/extractor-v1].
Action[https://example.com/actions/pref-extraction-001] schema:object Statement[@1].
Statement[@1] schema:confidence DecimalLiteral(0.82).
Statement[@1] prov:wasDerivedFrom CreativeWork[https://example.com/messages/msg-014].
```

**If `Claim` or `Assertion` were a first-class entity type**

This would let you separate the claim-like artifact from the underlying world relation.

```text
Claim[https://example.com/claims/alice-aisle-seat-like-001] schema:about Statement[@1].
Claim[https://example.com/claims/alice-aisle-seat-like-001] schema:confidence DecimalLiteral(0.82).
Claim[https://example.com/claims/alice-aisle-seat-like-001] prov:wasDerivedFrom CreativeWork[https://example.com/messages/msg-014].
Claim[https://example.com/claims/alice-aisle-seat-like-001] schema:author SoftwareAgent[https://example.com/agents/extractor-v1].
```

I would say this is possible today, but it wants a clearer official convention for “confidence attached to a statement.”

I would also be careful here: `Claim` may be a stronger candidate than `Preference`, but even then, it may still be better treated as a statement-level convention unless its identity rules are intentionally hardened.

## 5. Reasoning Trace

**Example in my own words**

“The agent handled a refund request and generated a multi-step reasoning trace for that task.”

**Best FCP modeling today**

I would model the overall trace as a `CreativeWork` if you mean the recorded trace artifact. If you mean the execution episode itself, I would usually model that as an `Action`. I think the best current split is:

- the reasoning episode = `Action`
- the recorded trace artifact = `CreativeWork`

That gives you both the process and the record.

**FSD-style sketch today**

```text
Action[https://example.com/actions/refund-resolution-run-001] schema:agent SoftwareAgent[https://example.com/agents/support-bot-v2].
Action[https://example.com/actions/refund-resolution-run-001] schema:object Concept[https://example.com/tasks/refund-request].
Action[https://example.com/actions/refund-resolution-run-001] schema:startDate DateTimeLiteral(2026-04-03T14:10:00Z).

CreativeWork[https://example.com/traces/refund-resolution-run-001] schema:about Action[https://example.com/actions/refund-resolution-run-001].
CreativeWork[https://example.com/traces/refund-resolution-run-001] schema:author SoftwareAgent[https://example.com/agents/support-bot-v2].
```

**If `ReasoningTrace` were a first-class entity type**

This would be cleaner for agent-memory systems.

```text
ReasoningTrace[https://example.com/traces/refund-resolution-run-001] schema:about Action[https://example.com/actions/refund-resolution-run-001].
ReasoningTrace[https://example.com/traces/refund-resolution-run-001] schema:author SoftwareAgent[https://example.com/agents/support-bot-v2].
ReasoningTrace[https://example.com/traces/refund-resolution-run-001] schema:name TextLiteral(Refund resolution trace).
```

This is workable today, but a first-class `ReasoningTrace` type would make it cleaner.

Still, under the Fide design rule, I would ask whether a reasoning trace is truly a new base kind or whether it is better understood as a specialized `CreativeWork` or graph of `Action` entities.

## 6. Reasoning Step

**Example in my own words**

“One step in the trace is: retrieve the customer’s order history.”

**Best FCP modeling today**

I would model a step as an `Action`.

That seems natural:
- it is attributable to an actor
- it has a target/object
- it can depend on previous steps
- it can happen at a time

**FSD-style sketch today**

```text
Action[https://example.com/actions/refund-step-1] schema:agent SoftwareAgent[https://example.com/agents/support-bot-v2].
Action[https://example.com/actions/refund-step-1] schema:name TextLiteral(Retrieve customer order history).
Action[https://example.com/actions/refund-step-1] schema:object Person[https://example.com/users/alice].
Action[https://example.com/actions/refund-resolution-run-001] schema:hasPart Action[https://example.com/actions/refund-step-1].
```

If you need sequencing:

```text
Action[https://example.com/actions/refund-resolution-run-001] schema:hasPart Action[https://example.com/actions/refund-step-2].
Action[https://example.com/actions/refund-step-2] schema:previousItem Action[https://example.com/actions/refund-step-1].
```

**If `ReasoningStep` were a first-class entity type**

```text
ReasoningStep[https://example.com/actions/refund-step-1] schema:name TextLiteral(Retrieve customer order history).
ReasoningStep[https://example.com/actions/refund-step-1] schema:agent SoftwareAgent[https://example.com/agents/support-bot-v2].
ReasoningStep[https://example.com/actions/refund-step-1] schema:object Person[https://example.com/users/alice].
ReasoningTrace[https://example.com/traces/refund-resolution-run-001] schema:hasPart ReasoningStep[https://example.com/actions/refund-step-1].
```

Again, works today, but conventions matter.

This is probably not a strong base-type candidate, because a reasoning step is very plausibly just an `Action` viewed in a particular application context.

## 7. Tool Call

**Example in my own words**

“The agent called `getOrderHistory` for Alice’s account.”

**Best FCP modeling today**

This fits `Action` very naturally.

The tool call is an action by a software agent directed at some resource, with arguments/result attached either as literals or related artifacts.

**FSD-style sketch today**

```text
Action[https://example.com/actions/tool-call-001] schema:agent SoftwareAgent[https://example.com/agents/support-bot-v2].
Action[https://example.com/actions/tool-call-001] schema:name TextLiteral(getOrderHistory).
Action[https://example.com/actions/tool-call-001] schema:object Person[https://example.com/users/alice].
Action[https://example.com/actions/tool-call-001] schema:startDate DateTimeLiteral(2026-04-03T14:10:03Z).
Action[https://example.com/actions/refund-step-1] schema:hasPart Action[https://example.com/actions/tool-call-001].
Action[https://example.com/actions/tool-call-001] schema:result JSONLiteral({"ordersFound":3}).
```

**If `ToolCall` were a first-class entity type**

This would mostly improve readability and standard downstream querying.

```text
ToolCall[https://example.com/actions/tool-call-001] schema:name TextLiteral(getOrderHistory).
ToolCall[https://example.com/actions/tool-call-001] schema:agent SoftwareAgent[https://example.com/agents/support-bot-v2].
ToolCall[https://example.com/actions/tool-call-001] schema:object Person[https://example.com/users/alice].
Action[https://example.com/actions/refund-step-1] schema:hasPart ToolCall[https://example.com/actions/tool-call-001].
ToolCall[https://example.com/actions/tool-call-001] schema:result JSONLiteral({"ordersFound":3}).
```

This one is already pretty good in FCP.

Under the stable-type rule, `ToolCall` is probably better kept as a modeling convention over `Action` unless Fide eventually decides that tool-invocation identity behaves differently enough to justify a new base type.

## 8. Observation / Publication / Provenance Timing

**Example in my own words**

“A statement was first publicly observable on GitHub at a particular time, while the underlying event may have happened earlier.”

**Best FCP modeling today**

This is actually one of the more powerful parts of your model, because FCP can cleanly distinguish:
- event time
- action time
- publication time
- observation time
- derivation/provenance time

You are already doing this in spirit with `prov:wasDerivedFrom`.

For statement-level timing, I would keep using the statement as subject.

**FSD-style sketch today**

```text
@1 Action[https://example.com/actions/refund-approved] schema:agent Organization[https://example.com/orgs/acme-support].
@2 @1 schema:startDate DateTimeLiteral(2026-04-03T14:12:00Z).

CreativeWork[https://example.com/docs/refund-audit-log-entry-55] schema:datePublished DateTimeLiteral(2026-04-03T14:15:00Z).
@3 @1 prov:wasDerivedFrom CreativeWork[https://example.com/docs/refund-audit-log-entry-55].
@4 @3 schema:dateCreated DateTimeLiteral(2026-04-03T14:15:00Z).
```

**If `Observation` were a first-class entity type**

This could help if you want a formal layer for externally witnessed existence.

```text
Observation[https://example.com/observations/github-observation-001] schema:about Statement[@3].
Observation[https://example.com/observations/github-observation-001] schema:observer Organization[https://example.com/indexers/github-proof-service].
Observation[https://example.com/observations/github-observation-001] schema:startDate DateTimeLiteral(2026-04-03T14:15:00Z).
```

For your anchor-ID world, this same pattern extends naturally to public observation evidence.

## What this says about FCP today

My honest read is:

**Already strong today**
- message as artifact vs text as literal
- event vs action
- tool call as action
- provenance links
- statement-level qualification

**Usable but wants conventions**
- preference
- confidence
- reasoning trace
- reasoning step
- session/conversation semantics

The protocol is not the blocker. The main need is probably blessed modeling conventions for recurring domains.

## The cleanest current modeling choices I would use

If I had to standardize a “best current practice” right now:

- conversation/session -> `Event`
- message/utterance artifact -> `CreativeWork`
- message text -> `TextLiteral`
- send/utter action -> `Action`
- preference -> ordinary statement, optionally qualified as a `Statement`
- confidence -> statement about a `Statement`, often via an evaluation `Action`
- reasoning trace execution -> `Action`
- reasoning trace record -> `CreativeWork`
- reasoning step -> `Action`
- tool call -> `Action`
- provenance/publication timing -> statements on the source `CreativeWork` or on the reified `Statement`

## The place where I think you should be strict

I would resist adding first-class types too quickly.

What I would add first is probably not new ontology, but a short canonical guide for:
- how to model conversations
- how to model messages
- how to model extracted claims and confidence
- how to model agent reasoning and tool calls

That would let the ecosystem converge before the vocabulary hardens.

## If You Eventually Add New Entity Types

The most defensible candidates from this list are:
- `ReasoningTrace`
- possibly a carefully-defined `Claim` or `Assertion` layer, if its identity rules are made explicit

The least necessary are:

- `Conversation`
- `Message`
- `Preference`
- `ReasoningStep`
- `ToolCall`

because those already sit fairly cleanly on `Event` and `CreativeWork`.
Or, in the case of `Preference`, on statement patterns and statement qualification.

The main question is not whether these can be modeled today.
They can.

The real question is whether enough interoperable systems will want to query these as first-class concepts often enough that the semantic convenience is worth hardening them into the vocabulary.
